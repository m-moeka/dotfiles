# PR レビュー支援 draw.io 図示スキル

## このスキルが解決する問題

AI が生成したコードのレビューで、差分だけを見ていると致命的な見落としが起きる：

- **差分に現れない修正漏れ** を見逃す（変更すべきだったのにしなかった箇所）
- **コードの構造を理解していない** ため、変更が適切かどうか判断できない
- **ユースケースの漏れ** に気づけない（どんな場面でこのコードが使われるか把握していない）

このスキルは「差分をきれいに見せる」のではなく、
**レビュアーが構造と設計意図を理解した状態でレビューに臨めるようにする** ことが目的。

## ワークフロー

### 1. PR 情報の収集と既存設計の理解

```bash
gh pr view <PR番号> --repo <owner/repo> --json title,body,state,author,baseRefName,headRefName,additions,deletions,changedFiles,files
```

PR の body と変更ファイル一覧から概要を掴んだ後、**差分を読む前に**変更対象の周辺を理解する。
差分だけ見ても構造上の漏れには気づけない。既存設計を理解して初めて「何が足りないか」が見える。

Explore エージェントを使って並行調査する：
- **型の階層と依存関係**: 変更対象の型がどこで定義され、何に使われているか
- **設計パターン**: このコードベースで確立されたパターン（例: Specification パターン、Repository パターン）
- **既存の類似機能**: 同じパターンで実装された既存機能（新機能の「お手本」になるもの）
- **呼び出し元・呼び出し先**: 変更されたコードを使っている/使われているコード

### 2. 重要な洞察の抽出

調査結果から、**レビュアーにとって最も重要な洞察**を見つける。
「問い」を立てて機械的に分割するのではなく、「この PR の本質は何か」「レビュアーが知らないと困ることは何か」を考える。

考えるべきこと：
- この PR の変更の本質を一言で言うと何か
- 差分に現れていないが、本来変更が必要だった箇所はないか
- 既存設計とどう整合しているか（あるいは不整合があるか）
- レビュアーが見落としやすいリスクは何か

### 3. draw.io で図を生成

`mcp__drawio__open_drawio_xml` ツールを使って図を表示する。

図の枚数や構成は PR の内容に応じて自由に判断する。
小さな PR なら1枚、大きな PR なら2-3枚。各図に明確な焦点を持たせる。

### 4. 補足分析: Explore エージェントによる多角的深掘り

ステップ3の図は主に静的構造と設計意図を捉える。
大規模・複雑な PR では、静的構造だけでは「この PR をマージして安全か」を判断しきれない。

**以下のいずれかに該当する場合、ステップ3の図を出力した後にこのステップを実行する:**
- 型の構造的リファクタリングを含む PR
- 同じパターンを複数のエンティティ/モジュールに適用する PR
- 50ファイル以上の変更を含む大規模 PR

小さな機能追加や bugfix ではスキップしてよい。

**実行方法:**
Explore エージェント（Task ツールの `subagent_type: "Explore"` ）を **並列で** 起動し、
以下の観点のうち PR に関連するものを調査させる。
各エージェントは1つの観点だけに集中し、深く調べる:

- **ランタイム挙動**: 変更されたコードの主要な呼び出し元を特定し、ユーザー操作→実行パスを追跡する。部分実装がある場合、未実装パスでの挙動を確認する。
- **横断的な完了度**: 同じパターンが適用されるべき全対象を列挙し、各対象の実装状態を比較する。
- **型安全性の境界**: ジェネリクス型パラメータの具体化/劣化ポイントを追跡する。as キャスト、any、string への劣化箇所を特定する。

すべての観点を毎回調査する必要はない。PR の性質に応じて、最も重要な 1-2 の観点を選ぶ。

**補足図の生成:**
Explore エージェントの調査結果を受けて、
「ステップ3の図では答えられない、レビュアーにとって最も重要な問い」に対して
補足図を 1-2枚 生成する。
図の形式やパターンは内容に応じて自由に選ぶ。色体系はステップ3の図と統一する。

## 図の設計原則（ステップ3・4共通）

### 原則1: 情報設計を優先する

図は「PRの情報を配置する場所」ではなく「レビュアーの理解を助けるストーリー」。
以下を意識する：

- **何を図にするかより、どう図にするかが重要**: 同じ情報でも、配置・ネスト・比較の仕方で伝わり方が全く違う
- **全体構造と具体例を両方示す**: 理解を促すために有用であると判断した場合は、全体のレイヤー構造の中に、具体的なフィールド名や値の例を自然に埋め込む
- **不要な情報は削る**: 図に載せないという判断も重要な設計判断

### 原則2: 視覚的な階層でストーリーを伝える

レビュアーは図をざっと眺めてから細部を読む。この2段階の読み方に対応する視覚設計をする。

**ざっと見てわかるレベル:**
- 色とサイズでグループ・関係性・重要度が読み取れる
- ネスト（箱の中の箱）で包含関係が視覚的に明確
- 矢印と配置でフロー・依存の方向が見える

**じっくり読むレベル:**
- 各ボックスにフィールド名、値の例、短い注釈がある
- 設計判断の「なぜ」が注釈で添えられている
- 差分外の関連コードが灰色点線で示され、確認理由が書かれている

### 原則3: 適切な図解パターンを選ぶ

PR の内容に応じて最も伝わる図解パターンを選ぶ。主なパターン：

**ネスト構造図**: エンティティの内部構造を示すとき。外側のボックスが全体、内側に属性・子要素を配置。色の濃淡でグループを表現する。
→ 例: CsvExportConfigureEntity の中に options, columns[], dataSource をネストして配置

**レイヤー図**: アーキテクチャの層構造を示すとき。上下または左右にレイヤーを並べ、矢印で依存方向を示す。
→ 例: 個別プロパティ定義(Layer1) → 共有型定義(Layer2) → エンティティ集約・インフラ(Layer3)

**比較カラム図**: パターンの同一性や BEFORE/AFTER を示すとき。複数カラムを横並びにし、`=` や `→` で対応関係を示す。
→ 例: schemaConfigure パターン = editorConfigure パターン = csvExportConfigure パターン

**実行フロー図**: 処理の流れを示すとき。ステップを横に並べ、矢印でつなぐ。各ステップに「何をするか」と重要な詳細を書く。

**BEFORE/AFTER 図**: 変更前後の構造比較。左右に並べ、変更点をハイライト。

これらを組み合わせて使う。1つのセクション内でネスト構造を使い、セクション全体で比較カラムを構成する、といった複合パターンも有効。

### 原則4: 具体性と概念性を融合させる

コードの転記は避けるが、レビュアーがPRを理解するために有用な場合は、具体的なフィールド名・型名・値の例を適切に組み込む。
「概念的な説明の中に具体性を溶け込ませる」のが理想。

**避けるべき書き方:**
```
┌─────────────────────────────┐
│ type PropertySpecification< │
│   TModelKey, TPropertyKey   │
│ > = {                       │
│   metadata: ...             │
│   schema: ...               │
│ }                           │
└─────────────────────────────┘
→ コードをそのまま箱に入れただけ。図にした意味がない。
```

**良い書き方:**
```
┌─ CsvExportConfigureEntity ──────────────┐
│                                          │
│  uuid: CsvExportConfigureUuid (branded)  │
│  version: number / name: 1-32文字        │
│                                          │
│  ┌─ options ────────────┐  ┌─ columns ─┐│
│  │ encoding: UTF-8 | …  │  │ Column[]  ││
│  │ delimiter: comma|tab  │  │ type別に  ││
│  │ lineBreak: CRLF|LF   │  │ 分岐     ││
│  └──────────────────────┘  └──────────┘│
│                                          │
│  WHY: テナント単位でCSV出力フォーマットを  │
│  カスタマイズ可能にするための設定エンティティ │
└──────────────────────────────────────────┘
→ ネストで構造が見え、具体値で実感が湧き、WHY で意図が分かる。
```

### 差分外の関連コードを明示する

レビュアーが「差分には出てこないが確認すべきコード」を特定できるようにする。
灰色点線の要素として描き、「なぜ確認が必要か」の理由を添える。

## draw.io 技術リファレンス

### 色体系

| 色 | fillColor | strokeColor | 用途 |
|---|---|---|---|
| 青 | `#dae8fc` | `#6c8ebf` | 既存コード（変更なし） |
| 黄 | `#fff2cc` | `#d6b656` | 既存コード（今回変更あり） |
| 赤 | `#f8cecc` | `#b85450` | 新規追加 |
| オレンジ | `#FFF3E0` | `#FFB74D` | 要注意/TODO |
| 灰色点線 | `#F5F5F5` | `#999999` + `dashed=1` | **差分外だが関連するコード**（要確認） |
| 白地 | `#FAFAFA` | `#E0E0E0` | 詳細/補足 |

同系色のグラデーションでグループ感を出すのも有効。
この色以外にも、レイヤーや構造を示すために適宜適切な色を用いて表現すること。

### テキストスタイル

フォントサイズ、太字や細字、色文字などを駆使し、メリハリのある図解にすること。

### レイアウト

- 図全体の幅: 1400px - 1800px
- セクション（A, B, C...）をグリッド状に配置
- 凡例は図の最下部に横一列で配置
- 各図にサブタイトルとして焦点を明示する

### XML の注意点

- **日本語は UTF-8 文字をそのまま使う**: `value` 属性内の日本語テキストは `\uXXXX` エスケープシーケンスではなく、実際の UTF-8 文字（例: `協力会社`）をそのまま記述すること。draw.io は Unicode エスケープを解釈せず、`\u5354\u529B` のようにそのまま表示してしまう（文字化けの原因）
- draw.io XML 内でダブルハイフンをコメントに含めない（XML パースエラーになる）
- HTML エンティティ: `<` → `&lt;`, `>` → `&gt;`, `&` → `&amp;`
- `value` 属性内で HTML タグ使用可: `&lt;b&gt;`, `&lt;br&gt;`, `&lt;font&gt;`

## 品質チェック

図を生成したら、以下を確認する：

- [ ] 各図に明確な焦点がある（サブタイトルで明示）
- [ ] 箱の中身が具体的（型名、フィールド名、値の例を含む）
- [ ] ネスト・レイヤー・比較など、適切な図解パターンを使っている（単純な箱+線だけになっていないか）
- [ ] 設計の「なぜ」が含まれている
- [ ] 差分外の関連コードが灰色点線で示されている（該当する場合）
- [ ] 凡例がある
- [ ] **この図だけを見て、PR の構造と設計意図が理解できるか？**
- [ ] ステップ4の条件に該当するか確認し、該当する場合は補足分析を実行した

## ユーザーへの出力

図を表示した後、以下をテキストで補足する：

1. **各セクションの要点**: 各セクションが何を示しているか簡潔に
2. **レビュー時の注目ポイント**: 特に確認すべき箇所とその理由
3. **差分外の確認推奨ファイル**: PR の差分に含まれないが、影響を受ける可能性があるファイルのリスト
