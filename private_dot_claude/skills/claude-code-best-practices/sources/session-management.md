# Session Management & 1M Context

1Mトークンのコンテキストウィンドウは長時間の自律動作を可能にするが、意図的に管理しなければコンテキスト汚染を招く。

## Context Rot（コンテキスト劣化）

コンテキストが増えるにつれ、注意が多くのトークンに分散し、古い無関係な内容がノイズになってモデルの性能が劣化する現象。
1Mコンテキストモデルでは **~300-400kトークン** 付近からある程度の劣化が観察されるが、タスク依存であり固定のルールではない。

コンテキストウィンドウはハードカットオフ。上限に近づくと、作業を要約して新しいコンテキストで継続する必要がある（= compaction）。

## Every Turn Is a Branching Point（ターンごとの選択肢）

Claude がターンを終えた後、次に取れるアクションは5つ：

1. **Continue** — 同じセッションで次のメッセージを送る
2. **Rewind** (`esc esc` / `/rewind`) — 過去のメッセージに戻ってやり直す。それ以降のメッセージはコンテキストから削除される
3. **Clear** (`/clear`) — 学んだことを蒸留して新しいセッションを開始
4. **Compact** (`/compact`) — セッションを要約して要約の上に継続
5. **Subagent** — 独立したコンテキストで作業を委任し、結果だけを受け取る

## When to Start a New Session

基本原則: **新しいタスクを始めるなら、新しいセッションも始める。**

グレーゾーン: 実装直後のドキュメント作成のように、コンテキストの一部がまだ必要な関連タスク。
ファイルを再読み込みするコスト vs コンテキスト劣化のコストを天秤にかける。
知能が強く求められないタスク（ドキュメント作成等）なら、効率のために既存セッションを続ける判断もあり。

## Rewind: 修正より巻き戻し

良いコンテキスト管理の最も重要な習慣は **rewind**。

典型例: Claude が5ファイルを読み、アプローチAを試して失敗。
- 悪い対応: 「うまくいかなかった、Xを試して」→ 失敗したアプローチの全コンテキストが残る
- 良い対応: ファイル読み込み直後まで rewind して「アプローチAは使わない。fooモジュールはそれを公開していない。直接Bで行って」

"Summarize from here" で Claude に学びを要約させ、ハンドオフメッセージを作ることもできる（未来の自分から過去の自分へのメッセージ）。

## Compact vs Fresh Session

**Compact** (`/compact`):
- モデルが会話を要約し、履歴を要約で置き換える
- ロッシー: 何が重要かは Claude の判断に委ねる
- 自分で何も書かなくてよい、Claude の方が網羅的に学びやファイルを含める可能性がある
- `/compact focus on the auth refactor, drop the test debugging` のように指示で方向づけ可能

**Clear** (`/clear`):
- 自分で重要な情報を書き出して完全にクリーンなスタート
- 手間がかかるが、何が関連するかを自分で決められる

## Bad Compact の原因

コンテキストロットにより、compaction が実行される時点ではモデルの知能が最も低い状態にある。

典型的な失敗: 長いデバッグセッション後に autocompact が発火 → デバッグの調査内容が要約される → 次のメッセージで「bar.ts のあの警告を直して」→ 要約からその警告が落ちている。

対策: 1Mコンテキストがあるので、autocompact に任せず **proactive に `/compact` を実行し、次にやることの説明を添える**。

## Subagents as Context Management

サブエージェントは、多くの中間出力を生むが最終的に結論だけ必要な作業に有効。

判断基準: **このツール出力にまた必要になるか、それとも結論だけでよいか？**

具体的なパターン:
- 「サブエージェントを立ち上げて、このスペックファイルに基づいて作業結果を検証して」
- 「サブエージェントで別のコードベースの認証フローを読んで要約して、それを参考に自分で実装して」
- 「サブエージェントで git の変更内容に基づいてドキュメントを書いて」
